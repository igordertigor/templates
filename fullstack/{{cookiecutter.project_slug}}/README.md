## First-time setup

### 1. Install dependencies

```bash
just install
```

### 2. Configure environment

```bash
cp .env.example .env
```

Edit `.env` and fill in all required values. At minimum:

- `POSTGRES_PASSWORD` — any strong password for local dev
- `AUTHENTIK_SECRET_KEY` — generate with:
  ```bash
  openssl rand -base64 60 | tr -d '\n'
  ```
{% if cookiecutter.shared_db == "n" %}
- `AUTHENTIK_POSTGRES_PASSWORD` — password for the Authentik Postgres instance
{% endif %}
- `MINIO_ROOT_PASSWORD` — any password for local dev
- `AUTHENTIK_CLIENT_ID` — **CRITICAL**: This must be set before starting Authentik. Generate with:
  ```bash
  openssl rand -hex 20
  ```
  This is used by the Authentik blueprint to configure the OAuth2 provider's Client ID.

### 3. Start backing services

```bash
just up
```

This starts PostgreSQL, Authentik, MinIO{% if cookiecutter.add_arq == "y" %}, and Redis{% endif %} via Docker Compose.

**IMPORTANT:** The `AUTHENTIK_CLIENT_ID` must be set in `.env` before starting Authentik. The blueprint uses it to configure the OAuth2 provider's Client ID. If this value is missing or empty, the blueprint will fail to apply and you'll see a "Client ID Error" when trying to authenticate.

{% if cookiecutter.shared_db == "y" %}
**Important:** If you see "database authentik does not exist" errors, it means the PostgreSQL volume already exists from a previous run. The init script only runs on **first volume creation**. Solutions:

**Option 1: Start fresh (if no important data):**
```bash
docker compose down -v  # Deletes all volumes
just up
```

**Option 2: Manually create the database:**
```bash
docker compose exec postgres psql -U postgres -c "CREATE DATABASE authentik;"
docker compose restart authentik-server authentik-worker
```
{% endif %}

### 4. Set up Authentik

Authentik is configured using an automated blueprint for local development. The blueprint creates the OAuth2/OIDC provider and application automatically.

#### Step 1: Complete Authentik initial setup

1. Visit http://localhost:8080/if/flow/initial-setup/ (note the trailing slash!)
2. Set a password for the default **akadmin** user
3. Log in at http://localhost:8080 with username `akadmin` and your password

#### Step 2: Verify blueprint was applied

The blueprint at `docker/authentik-blueprints/oauth2-application.yaml` should have automatically created:
- An OAuth2/OIDC Provider named "{{ cookiecutter.project_name }} Provider"
- An Application named "{{ cookiecutter.project_name }}" with slug `{{ cookiecutter.project_slug }}`

To verify:
1. Navigate to **Applications** → **Applications** in the Authentik admin UI
2. You should see "{{ cookiecutter.project_name }}" listed
3. Click on it to view details

#### Step 3: Retrieve the Client Secret

The client secret is generated by Authentik and must be retrieved from the admin UI:

1. Navigate to **Applications** → **Providers**
2. Click on "{{ cookiecutter.project_name }} Provider"
3. In the provider details, find the **Client Secret** field
4. Click the copy icon to copy the secret
5. **Save this securely** — you won't be able to see it again in plain text

**Note:** The client secret is sensitive. Don't commit it to version control.

#### Step 4: Test the integration

{% if cookiecutter.add_frontend == "y" %}
1. Start the development servers: `just dev`
2. Visit http://localhost:5173
3. Click "Sign in" or navigate to a protected route
4. You should be redirected to Authentik for authentication
5. After logging in, you'll be redirected back to the application
{% else %}
1. Visit the API docs at http://localhost:8000/docs
2. Click "Authorize" at the top
3. You should be redirected to Authentik for authentication
4. After logging in, you'll be redirected back to the API docs
{% endif %}

#### Understanding the Blueprint

The blueprint (`docker/authentik-blueprints/oauth2-application.yaml`) is automatically applied when Authentik starts. It:

- Uses the `AUTHENTIK_CLIENT_ID` from your `.env` file
- Creates an OAuth2/OIDC provider with secure defaults:
  - **Client Type**: Confidential
  - **Authorization Flow**: default-provider-authorization-implicit-consent
  - **Signing Key**: authentik Self-signed Certificate
  - **Access Token Validity**: 5 minutes
  - **Refresh Token Validity**: 30 days
- Configures redirect URIs for local development:
  - `http://localhost:5173/auth/callback` (frontend)
  - `http://localhost:8000/docs/oauth2-redirect` (API docs)

If you need to modify the configuration:
1. Edit `docker/authentik-blueprints/oauth2-application.yaml`
2. Restart Authentik: `docker compose restart authentik-server authentik-worker`
3. The blueprint will re-apply automatically

**Note:** Blueprints are idempotent. Re-applying the same blueprint won't create duplicates.

#### Setting up Client Credentials (Machine-to-Machine)

For machine-to-machine authentication, you'll need to set up a separate OAuth2 provider with client credentials flow:

1. **Create a Service Account User:**
   - Navigate to **Directory** → **Users** → **Create**
   - Username: `service-account-{{ cookiecutter.project_slug }}`
   - Name: `{{ cookiecutter.project_name }} Service Account`
   - Check **Service account** (this creates a non-interactive user)
   - Set a strong password (not used for OAuth2, but required by Authentik)

2. **Create a Client Credentials Provider:**
   - Navigate to **Applications** → **Providers** → **Create**
   - Choose **OAuth2/OpenID Provider**
   - Name: `{{ cookiecutter.project_name }} Service Provider`
   - Authorization flow: **Authorize Application (Implicit)**
   - Client Type: **Confidential**
   - Client ID: (auto-generated) — copy this for your service
   - Client Secret: (auto-generated) — copy this for your service
   - **Advanced protocol settings:**
     - Subject mode: Based on the User's hashed ID
     - Include claims in id_token: ✓
     - Issuer mode: Each provider has a different issuer
   - **Flow settings:**
     - Authentication flow: Leave empty (not used for client credentials)
     - Authorization flow: default-provider-authorization-implicit-consent

3. **Create an Application for the Service:**
   - Navigate to **Applications** → **Applications** → **Create**
   - Name: `{{ cookiecutter.project_name }} Service`
   - Slug: `{{ cookiecutter.project_slug }}-service`
   - Provider: Select the provider you just created
   - Policy engine mode: any

4. **Assign Permissions:**
   - Navigate to **Directory** → **Users**
   - Click on your service account user
   - Go to the **Groups** tab
   - Add the service account to appropriate groups (e.g., `authentik Admins` for full access)
   - Or create custom groups with specific permissions

5. **Test the Client Credentials Flow:**
   ```bash
   # Get an access token
   curl -X POST http://localhost:8080/application/o/token/ \
     -H "Content-Type: application/x-www-form-urlencoded" \
     -d "grant_type=client_credentials" \
     -d "client_id=<your-service-client-id>" \
     -d "client_secret=<your-service-client-secret>"
   ```

   Expected response:
   ```json
   {
     "access_token": "...",
     "token_type": "Bearer",
     "expires_in": 300
   }
   ```

6. **Use the Token:**
   ```bash
   # Call your API with the token
   curl http://localhost:8000/api/protected-endpoint \
     -H "Authorization: Bearer <access-token>"
   ```

{% if cookiecutter.auth_device_flow == "y" %}
#### Setting up Device Flow (for CLI/IoT devices)

Device flow is enabled by default in the OAuth2 provider created by the blueprint. Here's how to use it:

1. **Device Authorization Endpoint:**
   ```
   http://localhost:8080/application/o/{{ cookiecutter.project_slug }}/device/
   ```

2. **Initiate Device Flow from Your CLI:**
   ```bash
   # Request a device code
   curl -X POST http://localhost:8080/application/o/{{ cookiecutter.project_slug }}/device/ \
     -H "Content-Type: application/x-www-form-urlencoded" \
     -d "client_id=$AUTHENTIK_CLIENT_ID"
   ```

   Expected response:
   ```json
   {
     "device_code": "...",
     "user_code": "ABCD-EFGH",
     "verification_uri": "http://localhost:8080/activate",
     "verification_uri_complete": "http://localhost:8080/activate?user_code=ABCD-EFGH",
     "expires_in": 300,
     "interval": 5
   }
   ```

3. **User Activates Device:**
   - The user visits `verification_uri` (http://localhost:8080/activate)
   - Enters the `user_code` shown in the CLI
   - Authenticates with Authentik
   - Approves the device authorization

4. **Poll for Access Token:**
   ```bash
   # Poll the token endpoint (every 5 seconds as indicated by interval)
   curl -X POST http://localhost:8080/application/o/token/ \
     -H "Content-Type: application/x-www-form-urlencoded" \
     -d "grant_type=urn:ietf:params:oauth:grant-type:device_code" \
     -d "device_code=<device-code-from-step-2>" \
     -d "client_id=$AUTHENTIK_CLIENT_ID"
   ```

   While pending:
   ```json
   {
     "error": "authorization_pending"
   }
   ```

   After user approves:
   ```json
   {
     "access_token": "...",
     "token_type": "Bearer",
     "expires_in": 300,
     "refresh_token": "..."
   }
   ```

5. **Device Flow Configuration:**
   The blueprint configures the device flow with these settings:
   - **Device Code Validity**: 5 minutes
   - **Polling Interval**: 5 seconds minimum
   - **User Code Format**: 8 characters (XXXX-XXXX)

   To customize, edit the provider in Authentik admin UI:
   - Navigate to **Applications** → **Providers** → "{{ cookiecutter.project_name }} Provider"
   - Scroll to **Advanced protocol settings**
   - Adjust device flow settings as needed

#### Production / Stage Environments

For production and staging, you have two options:

**Option 1: Create environment-specific blueprints** (Recommended)
1. Copy `docker/authentik-blueprints/oauth2-application.yaml` to your infrastructure repo
2. Update the `redirect_uris_dev` context variable to match your environment:
   ```yaml
   context:
     redirect_uris_prod:
       - https://your-prod-domain/auth/callback
       - https://your-prod-domain/api/docs/oauth2-redirect
   ```
3. Deploy the blueprint to your production Authentik instance
4. Use Authentik's API or Terraform provider to manage blueprints in production

**Option 2: Manual setup per environment**
1. Repeat the application setup for each environment
2. Use environment-specific redirect URIs:
   - Stage: `https://your-stage-domain/auth/callback`
   - Prod: `https://your-prod-domain/auth/callback`
3. Update Kubernetes ConfigMaps and Secrets in `k8s/overlays/` with the environment-specific values

For production deployments, see the [Authentik Terraform Provider](https://registry.terraform.io/providers/goauthentik/authentik/latest/docs) for infrastructure-as-code management.

## Troubleshooting

### Authentik: "database authentik does not exist"

{% if cookiecutter.shared_db == "y" %}
If you see this error when starting Authentik, it means the `authentik` database wasn't created in the shared PostgreSQL instance. This can happen if you had an existing PostgreSQL volume before switching to Authentik.

**Solution: Manually create the database**
```bash
docker compose exec postgres psql -U postgres -c "CREATE DATABASE authentik;"
docker compose restart authentik-server authentik-worker
```

Alternatively, if you don't have important data yet:
```bash
docker compose down -v  # WARNING: Deletes all volumes!
docker compose up -d
```
{% endif %}

### Authentik blueprint not applying

If the OAuth2 application isn't created automatically:

1. **Check that `AUTHENTIK_CLIENT_ID` is set in your `.env` file:**
   ```bash
   grep AUTHENTIK_CLIENT_ID .env
   ```
   If empty, generate one:
   ```bash
   echo "AUTHENTIK_CLIENT_ID=$(openssl rand -hex 20)" >> .env
   ```
2. Check the Authentik worker logs for blueprint errors:
   ```bash
   docker compose logs authentik-worker | grep blueprint
   ```
   Common error: "Failed to apply blueprint: 'AUTHENTIK_CLIENT_ID' is not defined"
3. Restart Authentik to apply the blueprint with the corrected environment:
   ```bash
   docker compose restart authentik-server authentik-worker
   ```

For more details, see `docker/authentik-blueprints/README.md`.

### Authentik: "Client ID Error - The client identifier (client_id) is missing or invalid"

This error occurs when the OAuth2 provider doesn't have a valid Client ID configured. This typically happens when:

1. `AUTHENTIK_CLIENT_ID` was not set in `.env` before starting Authentik
2. The blueprint failed to apply due to the missing environment variable

**Solution:**
1. Stop Authentik:
   ```bash
   docker compose down
   ```
2. Ensure `AUTHENTIK_CLIENT_ID` is set in your `.env`:
   ```bash
   grep AUTHENTIK_CLIENT_ID .env
   ```
   If missing or empty, generate one:
   ```bash
   echo "AUTHENTIK_CLIENT_ID=$(openssl rand -hex 20)" >> .env
   ```
3. Start Authentik again:
   ```bash
   docker compose up -d authentik-server authentik-worker
   ```
4. Verify the blueprint applied successfully:
   ```bash
   docker compose logs authentik-worker | grep -i blueprint
   ```
   You should see: "Successfully applied blueprint"

### Authentik client secret not working

The client secret is auto-generated by Authentik and must be retrieved from the admin UI. It's **not** stored in the blueprint or `.env` file.

To retrieve it:
1. Log in to Authentik admin: http://localhost:8080
2. Navigate to **Applications** → **Providers**
3. Click on "{{ cookiecutter.project_name }} Provider"
4. Copy the **Client Secret** field

### Port conflicts

If you see "address already in use" errors:
- Authentik uses ports **8080** (HTTP) and **8443** (HTTPS)
- MinIO uses ports **9000** (API) and **9001** (Console)
- Backend uses port **8000**
- Frontend uses port **5173**
- PostgreSQL uses port **5432**

Check what's using a port:
```bash
lsof -i :8080  # Replace with the conflicting port
```

## Database migrations

After changing `backend/app/tables.py`:

```bash
# Generate a migration
just migration "describe your change"

# Review the generated file in backend/migrations/versions/
# Then apply it
just migrate

# Roll back one step if needed
just migrate-down
```

**Important:** Always review auto-generated migrations before applying. Alembic's
autogenerate is reliable for adding/removing columns and tables but may miss
complex constraints or indexes. Check the generated file before committing.

## Kubernetes deployment

### Prerequisites

- A running Kubernetes cluster (k3s, k3d, kind, or managed)
- [Flux CLI](https://fluxcd.io/flux/installation/) installed
- [kubeseal](https://github.com/bitnami-labs/sealed-secrets) installed
- Sealed Secrets controller running in your cluster:
  ```bash
  kubectl apply -f https://github.com/bitnami-labs/sealed-secrets/releases/latest/download/controller.yaml
  ```

### Seal your secrets

Before deploying, seal all secrets for each overlay. See the README in
`k8s/overlays/prod/sealed-secrets/` and `k8s/overlays/stage/sealed-secrets/`
for the exact commands.

```bash
# Example for postgres-secret in prod
kubectl create secret generic postgres-secret \
  --from-literal=username=postgres \
  --from-literal=password=<strong-password> \
  --namespace={{ cookiecutter.project_slug }}-prod \
  --dry-run=client -o yaml \
  | kubeseal --format yaml \
  > k8s/overlays/prod/sealed-secrets/postgres-secret.yaml
```

Add the sealed secret files to `k8s/overlays/prod/sealed-secrets/kustomization.yaml`.

### Flux bootstrap (one-time per cluster)

```bash
flux bootstrap github \
  --owner=your-org \
  --repository={{ cookiecutter.project_slug }} \
  --branch=main \
  --path=flux/clusters/my-cluster \
  --personal
```

After bootstrap, Flux reconciles everything under `flux/clusters/my-cluster/`
automatically. All further changes are made via Git.

### Updating image tags

Image tags are set in `k8s/overlays/prod/kustomization.yaml`. Update the
`newTag` field and push — Flux will roll out the new image:

```yaml
images:
  - name: {{ cookiecutter.project_slug }}-backend
    newTag: "abc1234"  # your image digest or tag
```

### Force immediate reconciliation

```bash
just flux-prod
{% if cookiecutter.use_stage_namespace == "y" %}
just flux-stage
{% endif %}
```
